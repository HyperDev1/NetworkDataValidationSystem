Create `server.py` in the project root. This is the container's HTTP entry point (separate from the CLI entry point `main.py`).

**Flask app setup:**
- Create Flask app instance
- Read port from `PORT` environment variable, default to 8080 (Cloud Run convention)
- Configure Python logging (same format as main.py: `%(asctime)s - %(name)s - %(levelname)s - %(message)s`)

**Health check endpoint — `GET /health`:**
- Returns JSON `{"status": "healthy"}` with HTTP 200
- No side effects, no config loading — must be fast and always succeed
- This is what Cloud Run uses to determine container readiness

**Validation trigger endpoint — `POST /validate`:**
- Loads Config() (which reads config.yaml from default path)
- Calculates date range: end_date = UTC now - 1 day, start_date = end_date - 7 days (same logic as run_single_validation)
- Calls `asyncio.run(run_validation(config, start_date, end_date))` — bridges async/sync
- On success, returns JSON: `{"status": "completed", "networks_processed": N, "failed": [...]}` with HTTP 200
  - Extract networks_processed from result dict (count of network keys)
  - Extract failed networks from result's `_failed_networks` list if present
- On exception, returns JSON: `{"status": "error", "message": str(e)}` with HTTP 500
- Wrap entire handler in try/except to ensure Cloud Run always gets an HTTP response

**Main block:**
- `if __name__ == '__main__':` runs `app.run(host='0.0.0.0', port=port, debug=False)`
- Use Flask's built-in dev server for simplicity (Gunicorn not needed — Cloud Run handles load balancing and concurrent requests are unlikely for this batch job)

**Update requirements.txt:**
- Add `Flask>=3.0.0` to requirements.txt (after existing dependencies)
- Do NOT add gunicorn — Flask dev server is sufficient per user decision context (Claude's discretion on Gunicorn vs Flask dev server)

<task type="auto">
  <name>Task 2: Create multi-stage Dockerfile and .dockerignore</name>
  <files>Dockerfile, .dockerignore</files>
  <action>
Create `Dockerfile` in the project root using a 2-stage multi-stage build:

**Stage 1 — Builder (named `builder`):**
```dockerfile
FROM python:3.13-slim AS builder
WORKDIR /build
# Install build dependencies needed for C extensions (pandas, pyarrow, aiohttp)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    && rm -rf /var/lib/apt/lists/*
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt
```

**Stage 2 — Runtime:**
```dockerfile
FROM python:3.13-slim
# Create non-root user (security requirement from CONTEXT.md)
RUN groupadd -r appuser && useradd -r -g appuser -d /app -s /sbin/nologin appuser
WORKDIR /app
# Copy only runtime dependencies from builder
COPY --from=builder /install /usr/local
# Copy application code
COPY main.py server.py service.py ./
COPY src/ ./src/
# Set ownership
RUN chown -R appuser:appuser /app
USER appuser
# Cloud Run convention: PORT env var
ENV PORT=8080
EXPOSE 8080
# Health check (Docker-level, useful for local testing — Cloud Run uses HTTP probe)
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8080/health')" || exit 1
# Entry point: Flask HTTP server
CMD ["python", "server.py"]
```

**Key decisions:**
- Use `--prefix=/install` in builder stage so we can COPY just the installed packages to runtime
- Non-root user `appuser` for security
- HEALTHCHECK for local Docker testing (Cloud Run ignores Docker HEALTHCHECK, uses its own HTTP probe)
- CMD uses python directly (not gunicorn)
- Do NOT copy config.yaml, credentials/, .git — they are excluded by .dockerignore AND not in COPY commands
